## 0. 一句话结论

ETF 智能投教平台是一个 **数据驱动、语义驱动、证据闭环** 的 ETF 投教辅助平台。

它不是普通 ETF 列表工具，也不是自动交易系统，而是通过接入 ETF 数据、指数信息、市场新闻、风险事件和投教规则，构建一个可解释、可追溯、可推演的 ETF 语义世界，让用户以更低门槛理解 ETF、比较 ETF、认识风险，并获得投教型辅助方案。

本阶段虽然是竞赛 Demo，但应按照“专业产品雏形”标准开发：

```text
速度是 Demo 级
界面是产品级
结构是可扩展级
表达是竞赛演示级
```

## 2. 产品定位

### 2.1 产品本质

ETF 智能投教平台的本质是：

```text
面向 ETF 投资教育场景的语义世界建模与投教辅助系统。
```

它通过以下方式提供价值：

```text
1. 将 ETF、指数、行业、风险、新闻事件、用户目标和证据来源转化为语义对象。
2. 将对象之间的关系转化为可视化语义图谱。
3. 将实时或准实时数据变化转化为风险信号、市场事件和场景推演输入。
4. 将投教规则和证据链绑定到 AI 生成的辅助方案中。
5. 将用户反馈用于修正用户画像和方案解释。
```

### 2.2 产品不是

本平台不是：

```text
1. 不是自动交易系统。
2. 不是收益预测系统。
3. 不是具体买卖推荐系统。
4. 不是券商投顾替代品。
5. 不是普通 ETF 数据列表页。
6. 不是单纯聊天机器人。
7. 不是一次性临时 Demo 页面。
```

### 2.3 产品应该呈现为

本平台应该呈现为：

```text
1. 专业金融科技 SaaS 工作台。
2. ETF 数据与新闻事件驱动的语义画布。
3. 面向新手用户的投教辅助工具。
4. 面向竞赛评委的 AI 实战成果。
5. 后续可接入 my_blueprint / Knowledge Hub / SWM 的业务应用样板。
```

## 3. Demo 阶段目标

### 3.1 Demo 核心目标

本阶段目标不是做完整商业系统，而是快速做出一个可以公网演示的专业产品雏形。

必须满足：

```text
1. 打开链接即可使用。
2. 不需要登录和注册。
3. 第一屏有完整产品质感。
4. 中间画布能体现“语义世界”。
5. 页面能体现“数据驱动”。
6. 能完成一条清晰演示路径。
7. 能展示投教方案、场景推演和证据链。
8. 明确标注 Demo 数据来源和合规边界。
```

### 3.2 Demo 演示重点

演示时重点体现以下 5 件事：

```text
1. 用户输入目标后，系统能形成用户画像。
2. 系统能构建 ETF 语义世界画布。
3. 系统能将 ETF 数据、新闻事件、风险因素和用户目标关联起来。
4. 系统能生成投教辅助方案，而不是直接荐股或交易。
5. 系统能展示方案依据、风险提示和证据链。
```

### 3.3 Demo 数据原则

Demo 阶段允许使用种子数据，但必须明确标注：

```text
data_source = demo_seed
```

页面必须展示：

```text
当前使用演示数据，仅用于功能展示。
真实 ETF 数据接口、指数数据接口和新闻数据接口已预留接入位。
```

如果后端模型未配置，AI 生成接口不得伪造真实模型结果，应返回：

```text
503 model_not_configured
```

前端显示：

```text
当前未配置真实模型，以下内容为演示数据。
```

## 4. 核心创新点

### 4.1 从“ETF 筛选器”升级为“ETF 语义世界”

传统 ETF 工具通常展示：

```text
名称、代码、涨跌幅、规模、费率、成交额
```

本平台展示：

```text
ETF 产品
→ 跟踪指数
→ 行业/主题暴露
→ 风险因子
→ 新闻事件
→ 用户目标
→ 投教方案
→ 证据链
```

这使用户不仅能看到数据，还能理解数据之间的意义关系。

### 4.2 从“AI 直接回答”升级为“AI 基于语义世界解释”

平台不让 AI 直接输出“买什么”，而是让 AI 基于：

```text
用户目标
ETF 基础数据
指数数据
市场指标
新闻事件
风险信号
投教规则
证据对象
```

生成投教型说明、风险解释和场景推演。

### 4.3 从“静态方案”升级为“场景推演”

平台支持用户点击不同场景：

```text
市场震荡
科技板块回撤
红利风格走强
投资周期缩短
风险偏好下降
流动性下降
```

系统展示：

```text
哪些 ETF 受影响
哪些风险因子变化
哪些新闻事件相关
方案稳定性如何变化
是否需要重新评估
```

### 4.4 从“结论展示”升级为“证据闭环”

任何投教结论都应绑定：

```text
数据来源
新闻来源
投教规则
风险说明
模型生成记录
证据对象
```

即使 Demo 阶段使用 demo_seed，也要在结构上保留证据链。

## 5. 核心用户与使用场景

### 5.1 目标用户

第一阶段面向：

```text
1. ETF 新手用户。
2. 想了解宽基、红利、科技、债券等 ETF 类型的年轻投资者。
3. 需要学习定投、风险分散、场景变化影响的投教用户。
4. 竞赛评委和演示对象。
```

### 5.2 用户痛点

```text
1. ETF 产品太多，不知道如何理解差异。
2. 只看到涨跌幅，无法理解背后的指数、行业和风险因素。
3. 新闻信息很多，但不知道会影响哪些 ETF 或主题。
4. 缺少适合新手的结构化解释。
5. 普通 AI 聊天容易直接给结论，缺少证据链和风险边界。
```

### 5.3 典型用户输入

```text
我每月想定投 1000 元，投资周期 3 年，风险偏好中等，想了解宽基、红利和科技 ETF。我是新手，希望方案不要太复杂。
```

系统应解析出：

```text
投资方式：定投
每月预算：1000 元
投资周期：3 年
风险偏好：中等
投资经验：新手
关注主题：宽基 / 红利 / 科技
偏好目标：学习优先、风险可控、方案简单
```

## 7. 数据驱动语义画布定位

### 7.1 画布定义

中间画布不是普通图谱，也不是纯前端静态关系图，而是：

```text
数据驱动的 ETF 语义世界画布。
```

它将多类数据统一转化为语义对象、语义关系、风险信号和证据节点。

### 7.2 画布数据来源

画布应预留以下数据来源：

```text
ETF 基础数据
ETF 市场数据
指数数据
行业/主题数据
持仓数据
财经新闻
政策信息
基金公告
投教规则
用户画像
AI 生成结果
```

### 7.3 Demo 阶段数据策略

第一阶段可使用 demo_seed 数据，但必须让前端呈现出真实数据接入位：

```text
ETF 数据状态
新闻数据状态
指数数据状态
风险信号数量
证据对象数量
最近更新时间
数据源类型
```

### 7.4 后续真实接入方向

后续真实数据接入时，数据流应为：

```text
ETF 数据接口
财经新闻接口
指数数据接口
公告/政策接口
        ↓
Data Connector
        ↓
Data Normalizer
        ↓
Semantic Extractor
        ↓
Risk Signal Builder
        ↓
Evidence Binder
        ↓
Semantic Graph Builder
        ↓
React Flow 前端画布
```

第一阶段前端只需要按这个结构预留状态和展示效果。

## 12. 合规边界

### 12.1 必须显示免责声明

平台所有方案区域、报告区域和导出内容必须显示：

```text
本平台内容仅用于投资教育和辅助分析，不构成任何投资建议、收益承诺或交易指令。市场有风险，投资需谨慎。
```

### 12.2 禁止输出内容

不得输出：

```text
明确买入建议
明确卖出建议
保证收益
收益率承诺
自动交易指令
个性化持仓建议
规避监管的表述
```

### 12.3 推荐表述方式

可以输出：

```text
该 ETF 类型适合进一步学习。
该主题在当前场景下波动风险较高。
该方案更偏向投教理解，不代表真实投资建议。
建议用户结合自身风险承受能力重新评估。
```

## 13. 与 my_blueprint / KH / SWM 的关系

### 13.1 Demo 阶段处理原则

本阶段不强接 my_blueprint、Knowledge Hub 和正式 SWM 服务。

但前端和接口命名应保留适配空间。

第一阶段使用：

```text
ETF Demo API
Demo Seed Data
Local Semantic Graph Adapter
```

后续升级为：

```text
my_blueprint WorkflowRun
Knowledge Hub Search / Evidence API
SWM Object / Relation / Action API
```

### 13.2 业务应用定位

ETF 智能投教平台未来可作为：

```text
业务应用层 Application Layer
```

my_blueprint 负责：

```text
执行用户画像解析、数据接入、语义建模、方案生成、场景推演和报告生成工作流。
```

Knowledge Hub 负责：

```text
ETF 知识资产、投教规则、新闻证据、风险规则、案例和评测数据的入库、检索、证据链与版本管理。
```

SWM 负责：

```text
ETF 语义对象、关系、状态、动作、证据绑定和反馈记录。
```

### 13.3 当前阶段只做轻量预留

第一阶段只要求：

```text
1. 组件命名不写死为 ETFGraph，而是 SemanticWorldGraph。
2. 图谱节点包含 source_system 字段。
3. 图谱节点包含 data_source 字段。
4. 图谱节点包含 source_ref 字段。
5. 接口命名采用 /api/semantic-world/graph。
```

不要求：

```text
1. 真实调用 KH。
2. 真实调用 my_blueprint。
3. 真实写入 SWM 正式表。
4. 实现完整 WorkflowRun / NodeRun 回写。
```

## 16. 第一阶段验收标准

### 16.1 视觉验收



## 19. 最终结论

ETF 智能投教平台第一阶段的目标不是做完整金融系统，而是做出一个专业、高级、可演示、可扩展的 AI 投教产品雏形。

它必须体现三个核心关键词：

```text
数据驱动
语义驱动
证据闭环
```

第一阶段先用 demo_seed 数据把效果做出来，但页面、组件、接口和概念必须为真实数据接入、my_blueprint 工作流、Knowledge Hub 证据链和 SWM 语义世界模型保留扩展空间。

最小成功标准是：

```text
评委打开链接后，能在 3 分钟内理解：
这是一个用 AI 和语义世界模型帮助用户理解 ETF、识别风险、查看证据、进行场景推演的专业投教平台。
```
